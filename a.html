Research & Strategy

Teardown to Strategy

How an examination of Claude Code's architecture reveals a blueprint for building AI application layers that compound — and why restaurants are the first deployment domain.

An examination of Claude Code's source code reveals a proven, production-grade architecture for building AI application layers — memory hierarchies, context management, permission systems, tool orchestration, and implicit feedback collection. The architecture is 70% domain-agnostic and 30% domain-specific. The hard problems have been solved. The blueprint is extractable.

The opportunity is applying that architecture to a domain where interaction volume and outcome clarity create a data flywheel: restaurant ordering via RCS messaging. Every order is simultaneously a customer served, a dollar earned, and a training tuple produced. The business that optimizes for order volume automatically optimizes for the most valuable asset in AI: labeled interaction data with measurable outcomes.

The long-term vision is not the product. It is the feedback environment — a system where operator corrections improve customer-facing behavior, customer outcomes refine operator recommendations, and the whole thing generates RLHF-grade training data as a byproduct of generating revenue. No competitor in the restaurant tech landscape is building this feedback loop. They're building features. The compounding customer memory graph and the density of cross-actor feedback signals are what make this defensible.

The repo is the map. The restaurant is the territory. The feedback environment is the treasure.

The Codebase

Claude Code v2.1.88 is a decompiled TypeScript application (~163,000 lines, 1,884 files) that ships as Anthropic's official CLI tool. The repository also contains a clean-room Python rewrite (~5,000 lines) reimplementing the core architecture.

What It Actually Is

This is the application/orchestration layer between the user and the Claude LLM. The model itself is a black box accessed via API. None of the model weights or inference code lives here. The entire codebase exists to answer three questions:

  1. What to send to the model — context assembly, system prompts, project memory, git state
  2. What to do with the model's responses — execute tool calls (Bash, FileEdit, Grep, etc.), loop results back
  3. How to present results to the user — React/Ink terminal UI
User input
  → Context assembly (system prompts, memory, git state, tool list)
    → Claude API call (streaming)
      → Tool execution
        → Loop back with tool results
          → Final response to user

Everything else — 163,000 lines of it — is edge case handling, memory management, permission systems, context compression, multi-agent coordination, and UI.

The Architecture That Matters

Agent Loop (query.ts, 785KB): The largest single file. Manages the cycle of user input → API call → tool routing → response.

Tool System (tools/, 45 directories): 40+ tools built via a buildTool() factory. Domain-agnostic abstraction.

Memory Hierarchy: Four timescales — immediate (transcript), short-term (compaction), medium-term (session/config), long-term (CLAUDE.md, auto-memory with nightly distillation).

Permission System: Ask/allow/deny with per-tool granularity, project scoping, enterprise override. An ACL system, not a chatbot feature.

Context Compaction: Three strategies — reactive, micro, trimmed. Emerged from a real constraint: naive truncation destroyed critical context.

Implicit Feedback Collection: Approve/deny signals, telemetry, cost tracking, transcripts. The deployment environment is simultaneously a data collection instrument.

Generalizable Insights

  1. The LLM is the easy part. The moat is the scaffolding.
  2. Context engineering > prompt engineering. What you feed matters more than how you phrase it.
  3. Memory needs multiple timescales.
  4. Treat agent actions like syscalls.
  5. Settings need a governance model.
  6. Patterns emerge from real constraints.
  7. AI application layers are closer to operating systems than web apps.

Technology is copyable. Accumulated memory isn't. The only thing that survives the "incumbents add AI" wave is data no single incumbent can assemble and network effects that compound independent of the AI layer.

What's Actually Defensible

  • Customer memory graph (strong, slow) — switching cost on the consumer side. Takes years.
  • Location knowledge (medium) — 18 months of learned behavior per restaurant.
  • Cross-restaurant data (strong, needs scale) — each restaurant makes every other better.
  • POS integration depth (medium) — 12–18 month pain-in-the-ass moat.
  • Google distribution (high leverage, fragile) — RCS showcase partner.

What's NOT a Moat

  • "Our AI is better" — same foundation models
  • "Our UX is better" — it's a text conversation
  • "First mover" — GrubHub was first. DoorDash won.

The RLHF Environment

4 concurrent feedback sources. 500K labeled interactions daily. Every order is a training tuple.

Most RLHF environments have one feedback source. This has four — customer taps, customer behavior, operator overrides, and outcome data — all labeling the same interaction from different angles. The cross-actor signal (operator correcting customer-facing behavior) is unique.

The Full Loop

Operator Console
  ↓ Playbook + permissions + corrections
Agent Layer
  ↓ Context assembly → Claude API → Tool execution → Feedback capture
RCS Client
  ↓ Customer interactions (orders, questions, corrections)
Feedback Aggregation
  ↓ (context, action, multi-source reward) tuples
Offline Training
  Reward model → Policy refinement → Personalization → Domain knowledge

Five Feedback Types

  1. Explicit Correction (strongest) — "No I said NO mushrooms."
  2. Accept/Reject (preference) — Upsell tapped Yes/No/ignored.
  3. Outcome (delayed reward) — Completed? Reordered? Tipped?
  4. Behavioral (implicit) — Response time, typed vs tapped, "?" signals.
  5. Cross-Actor (unique) — Operator overrides agent's customer-facing behavior.

The Claude Code source code showed us how to build an AI application layer that compounds. Restaurants are the deployment domain. The product is the ordering platform. The moat is the memory graph. The long-term value is the feedback environment no one else is building.

Everyone in the competitive landscape is building AI features — voice ordering, kiosk upsells, dashboard chatbots. These are features bolted onto existing products. They are commoditized the moment a POS company adds the same API call.

The opportunity is building an application layer that creates a direct feedback loop between AI and human behavior at scale — and happens to generate revenue while doing it.

What the Repo Taught Us

The Claude Code source code is not interesting because it's Claude Code. It's interesting because it is a reference implementation for how to build AI applications that get smarter because they get used. The architecture was extracted from real production constraints over millions of interactions. It works. And 70% of it transfers directly.

The Long-Term Vision

Year 1–2: Build the application layer. Operator intelligence as wedge, RCS ordering as platform. Generate revenue.

Year 2–3: The flywheel engages. Customer memory compounds. Cross-restaurant data enables personalization no single restaurant could achieve alone.

Year 3+: The feedback environment becomes the primary asset. Millions of (context, action, multi-source reward) tuples accumulate daily. The business generates revenue on every order. The feedback environment generates the training data that makes the next order better. These aren't parallel strategies — they're the same flywheel.

The product is the ordering platform. The moat is the customer memory graph. The long-term value is the feedback environment that no one else is building.
